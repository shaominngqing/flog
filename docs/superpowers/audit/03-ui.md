```yaml
id: UI-004
label: D
location: src/app.rs:146-175
title: NetworkState.filtered_indices caches with mutable laziness pattern
evidence: |
  pub struct NetworkState {
      filtered_indices: Vec<usize>,
      filter_dirty: bool,
  }
  
  pub fn filtered_indices(&mut self, store: &crate::domain::NetworkStore) -> &[usize] {
      if self.filter_dirty {
          self.filtered_indices.clear();
          for i in 0..store.len() {
              if let Some(entry) = store.get(i) {
                  if self.filter.matches(entry) {
                      self.filtered_indices.push(i);
                  }
              }
          }
          self.filter_dirty = false;
      }
      &self.filtered_indices
  }
  
  This is a classic "cache with dirty flag" pattern embedded in domain view logic. 
  The immutable return type with mutable interior causes confusion in call sites.
proposed_action: |
  Extract into a dedicated FilterCache type with explicit API:
  - struct FilterCache { indices: Vec<usize>, version: u64 }
  - impl FilterCache { fn get_or_rebuild(...) -> &[usize] }
  Or move to pure function: compute_filtered_indices(filter, store) -> Vec<usize>
  Phase 3 design should decide between:
  A) Pure function approach (rebuild on each access, no state)
  B) Explicit cache object with version tracking
  C) Delegate to domain layer FilterState
risk: low
```
